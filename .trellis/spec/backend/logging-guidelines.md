# 日志规范

## 当前约定

项目使用 Python 标准库 `logging`，按模块创建 logger：

```python
import logging

logger = logging.getLogger(__name__)
```

当前业务代码以缓存层为主要日志示例（`catalog/cache.py`）：Redis 读取、加锁或写入失败记录 `warning`，并继续走数据库降级路径。不要引入新的日志框架或自定义全局格式，除非先同步配置和规范。

## 日志级别

- `debug`：仅记录可帮助定位的降级/缓存竞争细节，例如缓存锁竞争；不记录请求正文。
- `info`：记录确实需要运营观察的生命周期事件；当前订单和通知主要依靠数据库状态，不要为每个正常请求增加 info 日志。
- `warning`：外部依赖暂时失败但系统可以继续服务，例如 Redis 故障并降级到数据库。
- `error`：无法恢复或需要人工关注的异常；保留异常对象并使用 `logger.exception` 记录堆栈。

## 应记录内容

日志应包含可定位的模块、操作和非敏感标识（如缓存名称、商品/订单内部 ID、降级原因），并说明是否采取了 fallback。示例：`catalog/cache.py` 的“Redis 读取失败，已降级数据库查询”。

## 不得记录内容

- JWT、refresh token、密码、Authorization 头和完整请求体。
- 用户手机号、详细收货地址、私信正文等个人或交易敏感数据。
- 将异常直接 `print()`；服务进程中的诊断必须使用模块 logger。
- 在循环任务中无界限打印每一条订单或消息，避免日志放大。

## 验证

新增日志时应检查异常路径仍然返回既有 API 契约，并为关键降级行为保留测试；日志不能替代业务状态、通知记录或审计数据。
