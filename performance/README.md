# 性能测试操作手册

本目录使用 Locust 对后端 API 进行压力测试。本文面向首次执行性能测试的开发者，按顺序操作即可完成一次可记录的实测。

> 注意：测试会创建大量用户、商品、订单和私信，并会实际创建订单。只允许连接本地开发数据库或独立测试数据库，严禁连接生产环境。

## 一、这项测试在测什么

性能测试不是检查页面是否能打开，而是模拟多个用户同时请求接口，记录服务器在持续压力下的响应速度和失败情况。

本项目的脚本会模拟三类行为：

- 未登录访客：浏览商品列表、搜索商品、查看商品详情。
- 已登录买家：查看买家订单、卖家订单和私信会话。
- 下单用户：对专门生成的一次性虚拟商品创建订单。

测试结果主要关注以下指标：

| 指标 | 含义 | 如何判断 |
| --- | --- | --- |
| 平均响应时间 | 一次请求平均花费的时间，单位是毫秒 | 越低越好，但不能只看这一项 |
| P95 | 95% 的请求不超过这个耗时 | 比平均值更能反映大多数用户体验 |
| P99 | 99% 的请求不超过这个耗时 | 用于观察少量极慢请求 |
| RPS | 每秒完成的请求数 | 相同压力下越高，吞吐能力越好 |
| 失败率 | 返回异常状态码或不符合预期的请求占比 | 正式结果应先确保为 0 或解释原因 |

例如，`商品详情 P95 = 80ms` 表示 100 次商品详情请求中，至少 95 次在 80 毫秒内完成；剩余最多 5 次可能更慢。

## 二、开始前确认

请在项目根目录 `secondhand-platform` 中打开 PowerShell，并确认以下条件：

1. 后端已启动，默认地址为 `http://127.0.0.1:8000`。
2. Redis 已启动。缓存测试依赖 Redis；Redis 不可用时接口会回退数据库，结果不能代表缓存优化效果。
3. 虚拟环境中已安装依赖；项目使用 `uv` 时可先执行 `uv sync`。
4. 后端设置为 `DEBUG=True`。性能数据生成命令会拒绝在 `DEBUG=False` 环境运行。
5. 当前数据库不是生产数据库。数据生成会写入大量测试数据。

可以先确认迁移已完成：

```powershell
.\.venv\Scripts\python.exe backend\manage.py migrate
```

## 三、先做一次小规模演练

第一次不要直接生成中档数据。先生成 `small` 档数据，确认账号、环境变量和压测脚本均正确。该档会生成约 100 用户、1,000 商品、500 订单和 2,000 私信。

```powershell
.\.venv\Scripts\python.exe backend\manage.py seed_performance_data `
  --profile small `
  --prefix perf `
  --password "PerfPass123!"
```

命令完成后会输出类似内容：

```text
性能数据生成完成。
测试账号邮箱：perf-buyer-0000@example.test
商品详情 ID：101,102,103,104,105,106,107,108,109,110
下单商品池：901,902,903,...
```

这里的数字每次都不同。请保留这段输出，下一步必须使用本次实际生成的邮箱和 ID，不能照抄示例值。

参数说明：

- `--profile`：数据量，可选 `small`、`medium`、`large`。
- `--prefix`：测试数据前缀。建议每次正式测试使用新前缀，例如 `perfmed01`，避免误用旧数据。
- `--password`：生成测试账号时使用的登录密码。

若命令提示已有相同前缀的数据，可更换 `--prefix`。确实需要删除该前缀的数据时，必须明确执行：

```powershell
.\.venv\Scripts\python.exe backend\manage.py seed_performance_data `
  --profile small `
  --prefix perf `
  --password "PerfPass123!" `
  --reset --confirm-reset
```

此操作会删除该前缀创建的测试订单、用户和分类，请确认前缀无误后再执行。

若只需要清理数据而不重新生成，可执行：

```powershell
.\.venv\Scripts\python.exe backend\manage.py seed_performance_data `
  --prefix perf `
  --cleanup-only --confirm-reset
```

## 四、配置本次测试

Locust 从 PowerShell 环境变量读取测试目标和测试数据。将以下命令中的邮箱、ID 替换为上一步命令打印出的真实内容，并在**同一个 PowerShell 窗口**执行：

```powershell
$env:PERF_BASE_URL = "http://127.0.0.1:8000"
$env:PERF_EMAIL = "perf-buyer-0000@example.test"
$env:PERF_PASSWORD = "PerfPass123!"
$env:PERF_DETAIL_LISTING_IDS = "101,102,103,104,105,106,107,108,109,110"
$env:PERF_WRITE_LISTING_IDS = "901,902,903,904,905"
```

各变量的作用如下：

| 变量 | 作用 | 填写来源 |
| --- | --- | --- |
| `PERF_BASE_URL` | 被测后端地址 | 本地通常为 `http://127.0.0.1:8000` |
| `PERF_EMAIL` | 登录压测账号 | “测试账号邮箱” |
| `PERF_PASSWORD` | 登录压测账号密码 | 生成数据时的 `--password` |
| `PERF_DETAIL_LISTING_IDS` | 匿名详情接口反复访问的商品 | “商品详情 ID”整行 |
| `PERF_WRITE_LISTING_IDS` | 创建订单使用的商品池 | “下单商品池”整行 |

环境变量只在当前 PowerShell 窗口有效。关闭窗口或重新打开终端后，需要重新设置。

## 五、运行测试

### 1. 冒烟测试

先运行 1 分钟、5 个虚拟用户的小测试。它的目的只是确认配置正确，不作为正式性能结论。

```powershell
.\.venv\Scripts\locust.exe -f performance\locustfile.py `
  --headless -u 5 -r 1 -t 1m
```

参数含义：

- `--headless`：不打开 Locust 网页控制台，直接在终端运行。
- `-u 5`：最多模拟 5 个并发虚拟用户。
- `-r 1`：每秒启动 1 个用户，5 秒后达到 5 个用户。
- `-t 1m`：总共运行 1 分钟。

命令结束后看终端统计表中的 `Failures`。正常情况下应为 `0`。若出现失败，请先查看失败请求的状态码：

- `401`：账号邮箱或密码错误，或认证接口未正常运行。
- `404`：商品 ID 不属于本次数据，或后端地址/API 路径不正确。
- `409` 或创建订单失败：下单商品已被使用；重新生成带新前缀的数据并更新商品池。
- 大量连接失败：确认后端确实运行在 `8000` 端口，并在浏览器访问 `http://127.0.0.1:8000/api/v1/listings/`。

冒烟测试出现失败时，不要把结果写入报告。修复配置后重新执行这一节。

### 2. 推荐使用一键测试

核心接口测试已提供一键脚本，无需再手工复制账号、商品 ID 或配置 Locust 环境变量。脚本会自动生成数据、预热缓存、执行读接口和订单创建测试，并解决登录与订单创建限流问题：

```powershell
powershell -ExecutionPolicy Bypass -File performance\run_performance_test.ps1
```

默认采用中档数据、20 个读接口用户、每秒启动 5 个用户、读接口持续 5 分钟。首次检查流程可使用小档数据：

```powershell
powershell -ExecutionPolicy Bypass -File performance\run_performance_test.ps1 `
  -Profile small -ReadUsers 10 -SpawnRate 1 -ReadDuration 1m
```

脚本会将最终中文报告与原始文件分开保存：

- `performance/测试结果/性能测试_日期时间/性能测试报告.md`：面向阅读的中文测试结果。
- `performance/中间文件/性能测试_日期时间/`：Locust 原始 CSV、HTML、预热数据和含短期 JWT 的测试配置。

原始中间文件不能提交到版本库或发送给他人。

## 六、冷缓存与热缓存怎么测

当前一键脚本默认会预热 60 秒后再记录读接口结果，即测量热缓存效果。若需比较“冷缓存”和“热缓存”：

1. 冷缓存：仅在独立测试 Redis 中清空缓存后，使用不含预热步骤的专项命令测试。
2. 热缓存：运行当前一键脚本，读取生成的中文报告中“商品详情”行。
3. 使用相同机器、数据规模和用户配置比较平均响应时间、P95、P99、RPS 和失败率。

不要通过重启后端来代替清空缓存，因为部分部署方式下 Redis 数据仍然存在。清空 Redis 只应针对专用测试 Redis 实例执行；若本地 Redis 还被其他项目使用，请不要执行全库清空。

严格的“缓存改造前 vs 缓存改造后”对比还需要：切换到缓存功能合入前的代码版本，在相同机器、相同数据库数据、相同 Locust 参数下运行一遍，再切回当前版本运行一遍。两次中任何一项不同，结果都只能作为参考，不能视为严格对比。

## 七、填写测试报告

正式运行后，可将 `性能测试报告.md` 中的真实数据填写到 `docs/performance-test-report.md`：

1. 记录测试日期、代码版本、电脑硬件、数据库、Redis 和 Python 版本。
2. 记录数据规模和生成命令，例如 `medium，prefix=perfmed01`。
3. 从中文报告找到对应接口行，填写平均响应时间、P95、P99、RPS 和失败率。
4. 保留“中间文件”目录中的 HTML 和 CSV，作为报告数据来源。
5. 只有相同环境下完成了“改造前”和“改造后”两次测试，才能在简历或项目说明中引用性能提升百分比。

建议每组正式测试至少运行两次。两次结果差异明显时，检查是否有其他程序占用 CPU、数据库是否在执行迁移或备份、Redis 是否被重启，然后重新测试。
